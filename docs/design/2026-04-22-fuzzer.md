# xrpl-confluence Fuzzer Design

**Date:** 2026-04-22
**Status:** M1 complete — smoke-tested 2026-04-22

## Goals

Three co-equal goals. The design deliberately treats them as three orchestration modes on top of shared infrastructure, not as three knobs on a single loop.

1. **Divergence detection.** Every node on the topology must agree on every ledger hash, every transaction result, and every state-metadata diff. Primary target: goXRPL vs rippled.
2. **Bug-finding.** Crash/panic/invariant-violation discovery in goXRPL (and secondarily rippled) via semantic and byte-level mutation.
3. **Consensus / network chaos.** Behavior under node restart, latency, partition, amendment flip — validating recovery and consistency.

## Non-Goals

- Performance benchmarking. Throughput is a side-effect measurement, not a goal.
- Replacing unit-level jtx (which already exists in goXRPL). The fuzzer operates at the network level.
- Testing the Clio/admin/WS surfaces. Submission is through standard peer/RPC interfaces only.

## Architecture

Two-layer split: fuzzer logic lives in the Go `sidecar/`; Kurtosis/confluence is the orchestrator.

```
xrpl-confluence/
  src/tests/fuzz.star              # Kurtosis wrapper: seed state, launch sidecar fuzzer, wire dashboard
  src/helpers/                     # seeding helpers (synthetic now, mainnet-snapshot later)
  sidecar/internal/fuzz/
    accounts/                      # deterministic account pool, tiered funding
    generator/                     # xrpl-go wrapper, amendment-aware tx selection & mutation
    oracle/                        # state-hash, tx-result, metadata-diff, invariant checkers
    corpus/                        # seeded RNG, corpus save/load, divergence reporter, shrinker
    runners/
      realtime.go                  # phase 1: real consensus, async tx submission
      controlled.go                # phase 2: standalone / driven close-per-step
      chaos.go                     # phase 2: realtime + chaos events injected
    cmd/fuzz/                      # CLI entrypoint; one binary, subcommand per runner
```

Rationale: fuzzer logic needs to be `go test`-able outside Kurtosis, reusable between runners, and independently releasable. Kurtosis owns only topology, networking, lifecycle.

**Starting point.** The existing `sidecar/cmd/trafficgen/` + `sidecar/internal/oracle/` + `sidecar/internal/rpcclient/` already provide: genesis-funded account pool, simple tx generators (Payment/TrustSet/OfferCreate/AccountSet), layer-1 state-hash oracle, and divergence fixture export. M1 refactors those into the layout above and extends them (xrpl-go, seeded RNG, layer-2 oracle). Oracle layers 1+ live in `internal/oracle/` (not `internal/fuzz/oracle/`) because that package already exists and is the natural home.

## Shared Infrastructure

### Account Pool (`accounts/`)

- Deterministic: all keypairs derived from `(runSeed, index)` via HKDF so any run is reproducible.
- ~50 accounts at phase 1, tiered:
  - **rich** — well above reserve, for arbitrary tx types
  - **at-reserve** — exactly at current reserve, for reserve-boundary bugs
  - **under-reserve** — temporarily below reserve (post-freeze), for edge cases
  - **multisig** — signer list installed
  - **regular-key** — regular key set, master disabled
  - **blackholed** — master disabled, no regular key
- Funded at genesis from a faucet account; tier transitions are an explicit setup step, not an inline concern of the generator.

### Tx Generator (`generator/`)

- Thin wrapper over xrpl-go. Does **not** re-implement tx encoding.
- Queries active amendments at startup via the `feature` RPC (or reads the `Amendments` ledger object) and builds a candidate set of `(txType, flags, fields)` combinations valid under that amendment state.
- Three generation modes, ratios configurable:
  - **well-formed** — xrpl-go built, valid fields, valid state prerequisites
  - **mutated-valid** — well-formed then one field mutated (out-of-range amount, wrong flag combo, etc.)
  - **random-bytes** — occasional raw-byte corruption of serialized tx for codec/validator fuzzing
- State-prerequisite awareness: a small dependency graph ensures TrustSet runs before IOU Payment, OfferCreate before OfferCancel, AMMCreate before AMMDeposit, etc. The generator picks only txs whose prerequisites are satisfied for the candidate sender.
- **xrpl-go version.** Pinned via `go.mod` (`xrpl-go@vX.Y.Z`). Every corpus entry stamps the active version so cross-version repros are unambiguous.

### Oracle (`oracle/`)

Five layered checks. Each check runs independently and can be disabled via config.

1. **State-hash** — after each closed ledger, fetch `ledger_hash` and `account_state_hash` from every node; divergence triggers a full report.
2. **Tx result** — for each submitted tx, compare result codes across nodes (a tx accepted by rippled but `tecUNFUNDED` on goXRPL is a bug).
3. **Metadata diff** — for each applied tx, diff `AffectedNodes` across nodes; structural difference is a bug.
4. **Invariants** — after each close: total XRP conservation, per-account reserve compliance, owner-count consistency, directory integrity.
5. **Shrinker** — on divergence, replay the last N txs and bisect to find the minimal reproducer. Run as an offline step, not during the main loop.

### Crash Detection

Covers goXRPL panics and rippled `SIGABRT`/`SIGSEGV`/assertion failures. Implemented as a service-health poller in the sidecar:

- Every close interval, query Kurtosis for each container's exit code.
- On non-zero exit, grep the last N log lines for `ASSERT`/`FATAL`/stack-trace markers and emit a bug-finding event carrying `(node, exit_code, stack_excerpt, seed, last_txs)`.
- Same mechanism for both implementations; no debugger attachment required.

### Corpus & Reproducibility (`corpus/`)

- Seeded RNG; every run's seed is logged.
- On divergence: auto-save `(seed, tx-sequence, pre-state, post-state-per-node)` under `corpus/divergences/<timestamp>/`.
- `cmd/fuzz replay <corpus-dir>` re-runs a saved sequence against a fresh topology for debugging.
- Future: corpus entries feed back into the generator as seed mutations (coverage-guided).

### Metrics & Dashboard

Extends the existing sidecar Prometheus surface with a `fuzz_*` namespace — no new endpoint.

- Counters: `fuzz_txs_submitted_total{tx_type,mode}`, `fuzz_txs_applied_total{tx_type,result}`, `fuzz_divergences_total{layer}`, `fuzz_crashes_total{node,impl}`
- Gauges: `fuzz_accounts_active`, `fuzz_corpus_size`, `fuzz_current_seed`
- Histograms: `fuzz_oracle_latency_seconds{layer}`, `fuzz_close_duration_seconds`

Confluence dashboard gains a "Fuzzer" panel reading those series. No changes needed in the dashboard server beyond adding the panel.

## Run Modes

### Phase 1: Real-time Runner (`runners/realtime.go`)

- Initially carries divergence + bug-finding. Chaos is a dedicated runner (`chaos.go`) added in phase 2 — the realtime runner is its base.
- Real consensus on the confluence topology; close timer is whatever rippled/goXRPL agree on (~4s).
- Fuzzer submits txs asynchronously via peer RPC; per-peer routing is explicit (supports "same tx to different peers at the same height" stress mode).
- Oracle layers 1-4 run after each observed close. Layer 5 runs post-run on the saved corpus.

### Phase 2: Controlled-close Runner (`runners/controlled.go`)

- Nodes in standalone mode (or consensus-timer override); fuzzer explicitly triggers ledger close.
- Used by divergence + bug-finding for deterministic, high-throughput runs and shrinking.
- Not suitable for chaos (consensus is bypassed).

### Phase 2: Chaos Runner (`runners/chaos.go`)

- Extends realtime runner with a chaos scheduler that injects events:
  - Node restart (SIGTERM + cold start)
  - Network partition (Kurtosis netsplit)
  - Latency injection (tc/netem)
  - Amendment flip mid-run (submit amendment tx, observe flag-ledger behavior across nodes)
- Chaos events are themselves corpus entries — a divergence during chaos is reproducible.

## State Seeding

**Phase 1: Synthetic.** The tiered account pool is funded from the standard rippled genesis account (`rHb9CJAWyB4rj91VRWn96DkukG4bwdtyTh`); no separate faucet setup. Pre-seeded objects (trust lines, a few offers, a small AMM) created via a deterministic setup script before the fuzz loop starts.

**Phase 2: Mainnet snapshot.** Load a real ledger state at height N, swap validator keys, reconcile amendment state, and fuzz on top. Separate module; does not affect phase 1 APIs.

## CI Profiles

Two profiles ship alongside M1; a third is reserved for later.

- **smoke** — 2 min, fixed seed, well-formed generation only, minimal topology. Runs on every PR that touches goXRPL or the fuzzer.
- **nightly** — 1–2 h, random seed, all generation modes, default topology. Runs on `main`.
- **release** — multi-hour, chaos runner enabled. Wired in once M4 lands.

Profiles are Kurtosis-arg presets, not separate binaries.
