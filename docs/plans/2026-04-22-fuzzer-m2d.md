# Fuzzer M2d Implementation Plan — Oracle Layers 3 & 4

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Raise the fuzzer's bug-detection bar beyond ledger-hash + tx-result agreement by adding (a) a per-tx metadata diff across nodes — diagnostic enrichment for layer-1 divergences — and (b) a single-node XRP conservation invariant that fails if total fuzzer-controlled balances ever increase across a ledger close.

**Architecture:** Extend `rpcclient.Client.Tx` so it returns the raw `AffectedNodes` along with the existing `TransactionResult`. A new `oracle.CompareTxMetadata` walks every node's metadata and reports structural differences. A new `oracle.InvariantPoolBalance` remembers the last observed sum of pool + genesis balances and fails if the next sum is strictly greater (XRP can only leave accounts as burned fees, so the sum must be monotone non-increasing). The realtime runner wires both checks alongside layers 1 & 2.

**Tech Stack:** Go 1.24, existing `rpcclient` / `oracle` packages; `encoding/json.RawMessage` for opaque metadata, `math/big.Int` for balance arithmetic.

**Branch:** `fuzz/m2d` stacked on `fuzz/m2b` (#10).

**Scope:**
- In: `rpcclient.Tx` returns AffectedNodes; `oracle.CompareTxMetadata`; `oracle.InvariantPoolBalance`; wired into realtime runner; corpus entries with `Kind: "metadata"` / `Kind: "invariant"`.
- Out (deferred): full cross-node AffectedNodes canonical-form comparison beyond byte-equality; owner-count invariant (needs `account_objects` walk per account, heavier); escrow/paychan-aware XRP sum (only matters once M2 ships those tx families).

```go
// TxResult holds the relevant fields from a `tx` RPC response.
type TxResult struct {
	TransactionResult string          `json:"transaction_result"`
	Validated         bool            `json:"validated"`
	AffectedNodes     json.RawMessage `json:"affected_nodes,omitempty"` // raw JSON, opaque to the helper
}

// Tx looks up a transaction by hash and returns its result code and metadata.
func (c *Client) Tx(hash string) (*TxResult, error) {
	raw, err := c.Call("tx", map[string]any{"transaction": hash})
	if err != nil {
		return nil, err
	}
	var wrapper struct {
		Meta struct {
			TransactionResult string          `json:"TransactionResult"`
			AffectedNodes     json.RawMessage `json:"AffectedNodes"`
		} `json:"meta"`
		Validated bool `json:"validated"`
	}
	if err := json.Unmarshal(raw, &wrapper); err != nil {
		return nil, fmt.Errorf("parse tx: %w", err)
	}
	return &TxResult{
		TransactionResult: wrapper.Meta.TransactionResult,
		Validated:         wrapper.Validated,
		AffectedNodes:     wrapper.Meta.AffectedNodes,
	}, nil
}
```

```go
package oracle

import (
	"bytes"
	"context"
	"encoding/json"
	"fmt"
)

// TxMetadataComparison is the cross-node diff for a single tx's AffectedNodes.
// Agreed is true iff every node returned byte-equal AffectedNodes JSON.
type TxMetadataComparison struct {
	TxHash   string         `json:"tx_hash"`
	Agreed   bool           `json:"agreed"`
	NodeMeta []NodeMetaBlob `json:"node_meta"`
	Errors   []string       `json:"errors,omitempty"`
}

// NodeMetaBlob is the AffectedNodes blob observed on one node.
type NodeMetaBlob struct {
	Name          string          `json:"name"`
	AffectedNodes json.RawMessage `json:"affected_nodes"`
}

// CompareTxMetadata fetches the tx on every node and compares the raw
// AffectedNodes JSON byte-for-byte. Because rippled emits AffectedNodes in a
// canonical order, byte-equality is a correct proxy for structural equality.
//
// This is primarily a diagnostic oracle: if layer 1 (ledger-hash) agrees,
// layer 3 must also agree by construction. When layer 1 reports a divergence,
// layer 3 tells you *which* state objects each node thinks the tx affected —
// making the reproducer substantially more actionable.
func (o *Oracle) CompareTxMetadata(ctx context.Context, hash string) *TxMetadataComparison {
	cmp := &TxMetadataComparison{TxHash: hash, Agreed: true}

	for _, n := range o.nodes {
		if ctx.Err() != nil {
			cmp.Agreed = false
			cmp.Errors = append(cmp.Errors, "ctx cancelled")
			return cmp
		}
		res, err := n.Client.Tx(hash)
		if err != nil {
			cmp.Agreed = false
			cmp.Errors = append(cmp.Errors, fmt.Sprintf("%s: %v", n.Name, err))
			continue
		}
		cmp.NodeMeta = append(cmp.NodeMeta, NodeMetaBlob{
			Name:          n.Name,
			AffectedNodes: res.AffectedNodes,
		})
	}
	if len(cmp.NodeMeta) < 2 {
		return cmp
	}
	ref := cmp.NodeMeta[0].AffectedNodes
	for _, nm := range cmp.NodeMeta[1:] {
		if !bytes.Equal(nm.AffectedNodes, ref) {
			cmp.Agreed = false
			break
		}
	}
	return cmp
}
```

```go
package oracle

import (
	"fmt"
	"math/big"

	"github.com/XRPL-Commons/xrpl-confluence/sidecar/internal/rpcclient"
)

// InvariantPoolBalance verifies that the sum of tracked account balances
// never strictly increases across consecutive CheckLedger calls.
//
// XRPL's canonical XRP conservation invariant states that total XRP in the
// system only decreases (via burned fees). Restricting the check to the
// fuzzer's pool + genesis account is a conservative proxy: the sum can only
// decrease from fees paid by our tracked accounts, never increase. A strict
// increase means either (a) rippled has a bug that mints XRP, (b) a node is
// returning stale/inconsistent data, or (c) our tracked set is wrong — all
// worth surfacing.
type InvariantPoolBalance struct {
	addresses []string
	lastSum   *big.Int
}

// NewInvariantPoolBalance tracks the given account addresses.
func NewInvariantPoolBalance(addresses []string) *InvariantPoolBalance {
	return &InvariantPoolBalance{addresses: addresses}
}

// CheckLedger sums current balances and compares to the previous sum. The
// first call always succeeds (establishes baseline). Subsequent calls fail if
// the sum strictly increased.
func (i *InvariantPoolBalance) CheckLedger(client *rpcclient.Client) error {
	sum := new(big.Int)
	for _, addr := range i.addresses {
		info, err := client.AccountInfo(addr)
		if err != nil {
			// Skip missing accounts silently — they may not yet exist at early
			// ledgers. A persistent lookup failure across many ledgers would
			// surface elsewhere (e.g. in submit errors).
			continue
		}
		b := new(big.Int)
		if _, ok := b.SetString(info.Balance, 10); !ok {
			return fmt.Errorf("%s: unparseable balance %q", addr, info.Balance)
		}
		sum.Add(sum, b)
	}
	if i.lastSum != nil && sum.Cmp(i.lastSum) > 0 {
		prev := i.lastSum.String()
		cur := sum.String()
		i.lastSum = new(big.Int).Set(sum)
		return fmt.Errorf("XRP conservation violated: prev=%s current=%s (increase of %s drops)",
			prev, cur, new(big.Int).Sub(sum, new(big.Int).SetBytes(new(big.Int).SetBytes([]byte(prev)).Bytes())).String())
	}
	i.lastSum = new(big.Int).Set(sum)
	return nil
}

// LastSum returns the last observed sum, or nil if CheckLedger has never been
// called. For logging / corpus entries.
func (i *InvariantPoolBalance) LastSum() *big.Int {
	if i.lastSum == nil {
		return nil
	}
	return new(big.Int).Set(i.lastSum)
}
```

**Note on the error math:** the error-message computation above double-converts and is wrong; simplify the increase calculation to `new(big.Int).Sub(sum, previousBigInt)`. When writing this out, hold `prev` as a `*big.Int` and compute `new(big.Int).Sub(sum, prev).String()` directly. The implementing agent should make this simple.

Cleaner version:

```go
if i.lastSum != nil && sum.Cmp(i.lastSum) > 0 {
	diff := new(big.Int).Sub(sum, i.lastSum)
	prev := i.lastSum.String()
	cur := sum.String()
	i.lastSum = new(big.Int).Set(sum)
	return fmt.Errorf("XRP conservation violated: prev=%s current=%s (increase of %s drops)", prev, cur, diff.String())
}
```

## Post-M2d Follow-ups

- **Owner-count invariant.** Walk `account_objects` per account; compare count to `OwnerCount` field. Catches reserve-accounting bugs.
- **Escrow/PayChan-aware XRP sum.** Once M2c+ adds those tx families, `InvariantPoolBalance` should sum escrow/paychan reserve amounts too — otherwise legit XRP movements (account → escrow) will falsely trigger the monotone-decrease violation when the escrow is released.
- **Initial-state snapshot + known lifetime-fee total.** A stronger invariant: sum(balances) + sum(escrow/paychan) + lifetime_fees = 100_000_000_000_000_000 drops. Needs per-tx fee tracking; cheap if we watch every `Fee` field in tx submissions.
- **Layer-3 canonical ordering.** If rippled ever changes its AffectedNodes emission order, byte-equality breaks. A canonical-JSON normalization step (sort keys, stable ordering) would be robust — worth it once we see a false positive.
